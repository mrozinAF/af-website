# Ascendance Foundry — Website Revamp
### Project Report for Leadership

*Prepared: July 2026*

---

## 1. Executive Summary

We rebuilt the Ascendance Foundry website from the ground up. The previous site was
built on WordPress and had begun to work against us: it looked dated, the interface felt
cluttered, and the homepage tried to serve two very different audiences — prospective
Fellows and prospective business clients — on a single page, leaving both confused about
where to go and what applied to them.

The new site is cleaner, more modern, and built around the people it serves. We replaced
generic stock imagery with real in-office photography, added video testimonials from our
Fellows, and split the crowded homepage into dedicated journeys so each audience now gets
content written specifically for them. Just as importantly, the site now runs on a modern
content system that lets our own team update text, photos, and videos without touching any
code. Every major decision in the project traces back to direct feedback we gathered from
our Fellows through a survey — this was a revamp driven by our community, not by guesswork.

---

## 2. Why We Did This

This project started by listening. We sent a survey to all of our Fellows asking for their
honest impressions of the old website, and we used their responses as the foundation for
every improvement that followed.

Three problems came through clearly. First, **aesthetics** — the site simply looked
outdated, and the interface felt cluttered and unpolished in a way that didn't reflect the
quality of the organization behind it. Second, **a homepage trying to do too much** — too
much information was crammed onto a single page. Third, and most damaging, **content that
didn't know who it was talking to** — Fellows and business clients were both landing on the
same homepage, which spoke to neither of them well. A prospective Fellow and a prospective
client want to know very different things, and the old site forced them to wade through
each other's content to find their own.

Rather than guess at fixes, we took the survey feedback and used it to brainstorm concrete
improvements. The result is a site that addresses each of those problems directly:
a cleaner look, a focused homepage, and separate, purpose-built paths for each audience.

## 4. How It Was Built

The project moved from idea to finished site through a fast, three-step process.

**Design.** We first created the look of every page as visual mockups using Claude Design.
This let us see and refine the actual appearance of each page — layouts, photography,
spacing, and style — before any of it was built for real. Designing visually up front meant
we could make decisions quickly and with confidence.

**Build.** Those approved designs were then built into the real, working website. Because we
started from concrete, agreed-upon designs rather than vague descriptions, the build was
both fast and faithful — the finished site closely matches what was designed, with no long
back-and-forth translating ideas into reality.

**Content system.** Finally, the site was connected to a modern content system (described in
the next section) so that all the text, photos, and videos are managed separately from the
design and can be updated by the team at any time.

The headline is speed *and* quality: designing visually first and building directly from
those designs let us produce a polished, custom result quickly, without the compromises that
usually come with off-the-shelf templates.

## 5. Content Management

The new site is powered by a content system called **Sanity**. It's worth explaining in
plain terms what this is and why it's a meaningful upgrade for the team.

Sanity is what's known as a **"headless" content system**. In practice, that means the
**content and the design are kept completely separate**. The words, photos, and videos live
in one place; the design and layout live in another. For the team, this has a very concrete
benefit: **non-technical staff can update the site's text, swap in new photos, or add a new
video without ever touching the code — and without any risk of accidentally breaking the
design.** You edit content in a friendly editor; the site takes care of presenting it
correctly. Published changes appear on the live site within about a minute.

The second advantage is flexibility. WordPress — the old system — pushes you into rigid,
pre-built templates: you shape your content to fit the template, whether or not it really
fits. Sanity works the other way around. We were able to build **custom content structures
that match exactly how our content actually works** — a "video testimonial" is genuinely a
video testimonial, a "team member" is a real profile with a photo and bio, a "job posting"
has its own fields for department and application link. The team fills in clearly labeled
fields, and everything lands on the page looking right by design.

In short: the team is no longer dependent on a developer for everyday content updates, and
the site's structure fits our organization rather than forcing our organization to fit a
template.

## 6. Results & Benefits

The revamp delivers several concrete gains for the organization:

**A cleaner, more credible presence.** The site now looks modern and polished, reflecting the
quality of the organization rather than undercutting it. First impressions matter, and this
is a far stronger one.

**Two clear audience journeys.** Fellows and business clients each have their own dedicated
space, with content written specifically for them. No more asking visitors to sort through
material meant for someone else.

**Authenticity that builds trust.** Real in-office photography and genuine video
testimonials from Fellows give the site a human, credible feel that stock photos and written
blurbs never could — a real asset for attracting both new Fellows and new clients.

**A site the team can run itself.** With the new content system, everyday updates — new team
members, new job openings, refreshed text and photos — can be handled in-house, quickly and
safely, without waiting on technical help.

**A foundation that fits us.** The underlying structure was custom-built around how our
content actually works, so it can grow and adapt with the organization over time.

## 8. Appendix — Technical Detail

### 8.1 Tech Stack

| Layer | Technology |
|---|---|
| Frontend framework | Next.js 15 (App Router), React 19 |
| Language | TypeScript |
| Content system (CMS) | Sanity (headless), project `xk0vwazc`, dataset `production` |
| Content editing UI | Sanity Studio (v6) with a custom desk structure |
| Images | Sanity image pipeline + `next/image`, served from `cdn.sanity.io` |
| Portable Text | `@portabletext/react` for rich-text rendering |
| Repository | pnpm monorepo — Studio at the root, frontend in `web/` |

**Notable constraint:** the frontend uses `next-sanity` **v9**, which predates the built-in
live-preview toolkit (`defineLive` / `SanityLive`, introduced in v11+). Live/draft preview and
Visual Editing are therefore **not** wired up. In its place, `web/src/sanity/live.ts` is a
lightweight `client.fetch` wrapper using time-based revalidation (60 seconds in production;
uncached in development). Enabling full draft preview would require a `next-sanity` upgrade
and a `SANITY_API_READ_TOKEN`.

### 8.3 How a Page Renders

1. `web/src/app/page.tsx` serves the homepage; `web/src/app/[slug]/page.tsx` serves every
   other page by its slug.
2. Both fetch `PAGE_QUERY` (`web/src/sanity/queries.ts`), a GROQ query that pulls a page and
   fully expands each content block, including nested images, calls-to-action, and referenced
   people/jobs.
3. `PageBuilder.tsx` reads the page's `content[]` array and renders each block with its
   matching React component in `web/src/components/blocks/`.

A block must be registered in three places to work end to end: the **schema**
(`schemaTypes/`), the page-builder list + registry in **`schemaTypes/index.ts`**, its
projection in **`queries.ts`** (if it has nested images/refs), and a case in
**`PageBuilder.tsx`**.

### 8.4 Content Model (Sanity Schemas)

**Document types:** `page` (generic page with a `content[]` page-builder array),
`person`, `jobPosting`, and a singleton `siteSettings` (navigation menu + nav button).

**Page-builder block types** (in `schemaTypes/blocks/`): `hero`, `pageHeader`,
`aboutMission`, `coreTeam`, `expertNetwork`, `expertsCarousel`, `fellowsIntro`,
`fellowVideos`, `offerLedger`, `manifestoHeader` *(For Businesses)*, `capabilities`,
`whyPartner`, `textSection` *(Vision)*, `diptych` *(Vision)*, `compareColumns` *(Vision)*,
`comingSoonHero` *(Summer Camp)*, `pathCards`, `fellowshipDetail`, `accordion`,
`videoTestimonials`, `peopleGrid`, `jobListings`, `ctaBanner`, plus preset blocks
`richText`, `imageBlock`, and `cta`.

*(Page-section labels above in italics are taken from comments in the code that tie blocks to
specific pages — corroborating the built page set, including For Businesses and Vision.)*

**`person`** — team members and Fellows:

```ts
fields: [
  { name: 'name',        type: 'string', required },
  { name: 'role',        type: 'string', required },
  { name: 'photo',       type: 'image',  hotspot },
  { name: 'credentials', type: 'text' },   // "Co-founder of X; Harvard PhD"
  { name: 'bio',         type: 'text' },
  { name: 'order',       type: 'number' }, // left-to-right order in team grid
  { name: 'linkedIn',    type: 'url' },
]
```

**`jobPosting`** — open roles listed on the careers page:

```ts
fields: [
  { name: 'title',       type: 'string', required },
  { name: 'department',  type: 'string' },
  { name: 'description', type: 'text' },
  { name: 'applyLink',   type: 'url' },
  { name: 'datePosted',  type: 'date' },
]
```

**`videoTestimonials`** — the Fellows-page video block. Each testimonial accepts either an
uploaded video file **or** a YouTube/Vimeo/MP4 link, with a name, role, and optional quote;
validation requires at least one video source per testimonial:

```ts
{ name: 'title', type: 'string' }, { name: 'intro', type: 'text' },
videos: [{
  name:  'string' (required),
  role:  'string',
  videoFile: 'file (video/*)',        // upload an .mp4/.mov
  videoUrl:  'url',                   // ...or paste a YouTube/Vimeo/.mp4 link
  quote: 'text',
}]  // custom rule: each item must have a videoFile or a videoUrl
```

**`siteSettings`** — singleton driving the header/footer navigation: an ordered list of
`{ label, page-reference }` items, plus an optional call-to-action button (e.g. "Apply").
The frontend falls back to a hardcoded menu in `web/src/components/nav.ts` if navigation
isn't configured.
